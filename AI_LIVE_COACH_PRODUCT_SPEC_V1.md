# Speak Up AI Live Coach Product Spec V1

## 文档目的

这份文档定义训练页右侧 `AI Live Coach` 的目标产品形态。

重点解决 3 个问题：

1. 当前右侧面板不应该继续做“滚动事件流”
2. 右侧面板应该稳定呈现固定维度，而不是被单条 insight 抢占
3. 每个维度应该有哪些细粒度指标、状态定义、数据来源和刷新节奏

---

## 一、先修正产品定位

### 当前问题

当前右侧 `AI Live Coach` 更像“另一种字幕流”：

- 不同来源的 insight 在同一个列表里竞争
- 单条提示会不断刷屏
- 用户只能看到“最新一句话”，看不到稳定的整体状态
- 一次误报会直接污染主卡片

这不符合“演讲训练教练”的产品形态。

### 目标产品形态

右侧 `AI Live Coach` 改成 **固定维度状态面板**，而不是滚动 feed。

推荐形态：

1. 顶部：`当前最值得纠正的一条建议`
2. 中部：固定 3 张维度卡片
3. 底部：仅在内部诊断视图下展示调试信息

固定 3 个维度：

1. `肢体 & 表情`
2. `语音语调 & 节奏`
3. `内容 & 表达`

### 三个维度的评估边界

这 3 个维度需要给模型明确边界，但不能写成机械 checklist。

推荐口径：

#### 1. 肢体 & 表情

模型主要看这些：

- 入镜是否稳定、是否居中
- 头肩和上身是否端正、是否明显歪斜或松垮
- 整体是自然打开、平和稳定，还是偏紧、偏收、偏僵
- 手势是否自然参与表达，是否过少、过乱，或和内容不同步
- 是否存在明显的即时坏习惯，例如托腮、长时间低头、持续偏头、明显离镜
- 如果脸部和眼神足够清楚，再看是否有互动感、是否过于严肃或紧张

同时给模型这些约束：

- 如果当前只看得到头部，或者画面证据不足，不要对完整体态下结论
- 不要因为单帧表情或一次瞬时动作就给强结论
- 这一维度优先反映“当前正在发生的身体问题”，不要反复重复几秒前的旧问题

#### 2. 语音语调 & 节奏

模型主要看这些：

- 语速是偏快、偏慢，还是基本合适
- 停顿是否自然，重点句前后是否有停顿空间
- 吐字是否清楚，是否有明显卡壳、重复起头、找词感
- 语气是偏平、偏紧、偏虚，还是比较稳、有起伏
- 是否有明显重音和节奏变化，关键句有没有被讲出来
- 声音状态是有支撑、平和，还是明显发虚、发紧、发飘

同时给模型这些约束：

- 不要因为一句疑问句、反问句、试探性表达，就直接判断“声音不够有力量”
- 只有在最近一小段里持续出现低能量、发虚、节奏散的问题，才给强提醒
- 不要把正常思考停顿误判成严重卡壳

#### 3. 内容 & 表达

模型主要看这些：

- 这一段是否够简洁，是否绕得太远
- 观点是否推进清楚，听众是否容易跟住
- 结论是否足够靠前，还是铺垫太久
- 是否有明显重复、空转、解释过度
- 口头禅 / 填充词是否开始影响专业感
- 过渡是否自然，是否从一句到下一句还能接得上

同时给模型这些约束：

- 这一维度不判断“事实对错”，只判断表达是否清楚、利落、好跟
- 不要因为一句话稍长，就立刻上升成问题
- 只有当重复、口头禅、绕圈在最近一段里比较明显时，才给 `优先调整`

#### 总体原则

- 每次更新时，模型应优先抓每个维度里最值得提醒的一件事，不要在单个维度里一次塞很多问题
- 如果证据偏弱，就给 `分析中` 或 `基本稳定`，不要硬给强结论
- 如果当前没有明显坏点，可以给轻量正向反馈，但不要为了夸而夸
- 顶部主建议应该优先来自“当前最重要、最能立刻调整的一件事”
- 先判断这一小段表达的情绪与意图，再判断肢体、表情、语音是否匹配
- 不要默认用“中性演讲姿态”去要求所有场景；激动、兴奋、庆祝、低沉、严肃都可能是合理表达
- 只有当肢体或语音明显破坏了当前情绪表达、听感清晰度或观感稳定性时，才应该上升成问题

### 评估文案约束

这里要额外限制模型的表达方式。

产品不接受这类过于抽象、不可执行的判断：

- `表达略显犹豫`
- `语气更有力量一点`
- `手放下别乱比划`
- `内容再自然一点`

这类话的问题是：

- 用户不知道模型到底看到了什么
- 用户不知道下一步具体怎么改
- 不同维度之间容易出现空泛、重复的提醒

推荐统一写成：

- `可观察到的现象`
- `一个立刻可执行的动作`

例如：

- 不说 `表达略显犹豫`
- 改成 `这一句有重复起头，直接把结论说完`

- 不说 `语气更有力量一点`
- 改成 `重点句尾别收掉，最后四个字压实`

- 不说 `手放下别乱比划`
- 改成 `先把手离开脸，手势收回胸前`

- 不说 `内容不够自然`
- 改成 `先讲结论，再补原因`

### 提示词落地规则

实际写进模型提示词时，建议直接按子类展开，而不是只写三个大维度名字。

推荐映射方式：

- `肢体 & 表情`
  按 `入镜与构图 / 体态与对齐 / 肢体松弛度与紧绷感 / 手势自然度与连贯性 / 走位与空间使用 / 表情与眼神互动` 这 6 组来约束模型

- `语音语调 & 节奏`
  按 `吐字与清晰度 / 音量与投射感 / 语速 / 断句与停顿 / 重音与强调 / 起伏与情绪状态 / 卡壳与流畅度` 这 7 组来约束模型

- `内容 & 表达`
  按 `简洁度 / 口头禅与冗余 / 重复与绕圈 / 逻辑结构 / 观点清晰度 / 例子与支撑 / 收束与推进` 这 7 组来约束模型

同时要给模型这几个硬约束：

- 每次每个维度只抓一件最重要的事
- 优先抓听众最容易感知、且下一句最容易调整的点
- 证据不够时宁可保守，不要硬判
- 不允许输出抽象人格判断，必须写成“信号 + 动作”

### 三个维度的具体判定口径

这里不是让模型机械打勾，而是给它更清楚的边界。

#### 肢体 & 表情

优先识别这些更具体的可见信号：

- 人是否明显偏离镜头中心
- 是否只有头部可见，还是头肩、上半身已经稳定进入画面
- 头是否持续偏向一侧
- 肩线是否持续歪斜
- 是否出现托腮、摸脸、拿东西挡脸、长时间低头
- 手势是完全缺失、基本自然，还是过于频繁并分散注意力
- 如果是明显的情绪高点或强调句，短暂的上举拳、上扬手势、向上的强调动作，可以视为正向支撑动作，不要默认当成问题
- 如果这一小段明显是激动、兴奋、庆祝、强强调的表达，更大的手势、更亮的表情、更有冲劲的动作都可能是合理表现
- 如果这一小段明显是低沉、严肃、反思、克制的表达，更收一点、更稳一点、更低能量的肢体与表情也可能是合理表现
- 如果是全身或半身画面，再看站姿是否飘、重心是否反复晃、走位是否自然
- 如果脸部够清楚，再看表情是否过于僵、紧绷，是否有基本互动感

不要轻易下这类判断：

- `肢体不自然`
- `表情不太对`
- `状态一般`

应该改成更具体的口径：

- `身体偏左，先回到镜头中间`
- `头一直偏着，先把头肩摆正`
- `手一直放在脸边，先把手离开脸`
- `手势有点碎，先把动作收小一点`

#### 语音语调 & 节奏

优先识别这些更具体的可听信号：

- 这句是否有明显重复起头
- 如果是明显的强调式重复，例如 `very very nice`、`太棒了太棒了` 这种情绪放大，不要直接判成“重复起头”或“卡壳”
- 如果这一小段明显是激动、兴奋、庆祝式表达，局部更高的能量、更快一点的强调、更重的重音，未必是问题
- 如果这一小段明显是低沉、严肃、反思式表达，更平一点、更慢一点、更收一点，也未必是问题
- 是否连续出现 `嗯 / 哦 / 然后 / 就是` 这类口头禅 / 填充词
- 是否有明显卡壳、半句停住、重新起句
- 语速是否持续偏快，而不是某一句稍快
- 重点句前后是否完全没有停顿空间
- 句尾是否经常收掉，导致重点落不下来
- 吐字是否发虚、含混，还是基本清楚
- 整体声音是平稳、有支撑，还是持续发紧、发飘、发虚

不要轻易下这类判断：

- `表达略显犹豫`
- `语气不够有力量`
- `节奏不太自然`

应该改成更具体的口径：

- `这句重复起头，直接把后半句说完`
- `口头禅有点多，先把“然后”拿掉`
- `语速连续偏快，这一句放慢半拍`
- `重点句后别立刻接下一句，先停一下`
- `句尾别收掉，把最后几个字讲完整`

#### 内容 & 表达

优先识别这些更具体的表达信号：

- 是否先铺垫太久，还没有说到结论
- 是否一句里装了太多层意思
- 是否同一个点重复讲了两遍以上
- 是否在绕圈解释，没有明显推进
- 是否口头禅 / 填充词已经开始影响专业感
- 这一段是否能让听众轻松跟上

不要轻易下这类判断：

- `内容不够自然`
- `表达有点绕`
- `逻辑可以更清楚`

应该改成更具体的口径：

- `先讲结论，再补原因`
- `一句只讲一个点`
- `这层铺垫先删掉`
- `这个点重复了，往下推进`
- `先把主语和结论说出来`

## 二、右侧面板的最终信息架构

### 1. 顶部主建议

作用：

- 只显示当前最值得用户立刻调整的一件事
- 不滚动、不累计历史
- 始终保持一句话标题 + 一句行动建议

示例：

- `把头肩位置先定住`
- `这一段内容清楚，但镜头感偏弱，先把头肩摆正，再继续讲重点句`

### 2. 三张固定维度卡

每张卡都固定包含：

- 维度名
- 当前状态
- 当前判断
- 当前建议

不做滚动历史。

### 3. 内部诊断区域

只在内部排障或开发联调时展示：

- 原始 provider 事件
- 维度状态的内部信号
- 聚合前后的状态变化

默认用户不看。

## 三、统一状态模型

### 先区分两层状态

这里必须区分：

1. **内部状态机**
2. **用户可见文案**

之前的问题在于把内部状态直接暴露给用户了。

这是不对的。

像：

- `positive`
- `neutral`
- `warning`
- `insufficient_signal`

这些词可以用于工程内部，但**不应该直接出现在用户界面上**。

尤其：

- `insufficient_signal`

这是模型/规则链路内部的判断条件，不是产品语言。用户看到这个词，只会困惑。

### 内部状态机

内部可以继续保留 4 档：

- `positive`
- `neutral`
- `warning`
- `insufficient_signal`

说明：

- `positive`：当前表现明显可取
- `neutral`：当前没有大问题，但没有明显亮点
- `warning`：当前存在值得优先调整的问题
- `insufficient_signal`：当前信号不足，不能下可靠结论

### 用户可见状态

前端不要直接显示上面 4 个英文标签。  
用户真正看到的应该是下面这套更自然的产品语言：

- `做得好`
- `基本稳定`
- `优先调整`
- `分析中`

对应关系：

- `positive -> 做得好`
- `neutral -> 基本稳定`
- `warning -> 优先调整`
- `insufficient_signal -> 分析中`

### 为什么用户侧应该用“分析中”

因为用户不需要知道内部判定条件，也不需要知道当前为什么还没形成结论。

用户只需要知道一件事：

- 这一项反馈还在更新中

所以用户侧统一用：

- `分析中`

不要展示：

- `insufficient_signal`
- 任何“为什么暂时没结论”的解释

### 典型场景

例如：

- 只露一个头，却去判断肩线倾斜
- 手完全不在画面里，却去判断手势自然度
- 全景不可见，却去判断走位

这些情况都应该进入：

- 内部：`insufficient_signal`
- 用户侧：`分析中`

## 四、维度一：肢体 & 表情

这个维度不是纯“姿态”，而是覆盖：

- 镜头中的身体状态
- 肢体表达状态
- 面部状态
- 与观众的视觉互动感

### 4.1 一级子类

建议拆成 6 个子类：

1. 入镜与构图
2. 体态与对齐
3. 肢体松弛度与紧绷感
4. 手势自然度与连贯性
5. 走位与空间使用
6. 表情与眼神互动

### 4.2 细粒度指标清单

#### A. 入镜与构图

指标：

- 是否完整入镜
- 头部是否被切边
- 肩膀是否稳定入镜
- 上半身是否足够可见
- 是否居中
- 镜头距离是否合理
- 当前是 `head-only / upper-body / full-body`

用户侧文案例子：

- `当前入镜比较稳定，镜头交流感不错`
- `你的人物位置略偏左，回到镜头中心会更有交流感`

当前数据来源：

- 前端 pose
- 后端 pose rule
- Omni 做补充解释

#### B. 体态与对齐

指标：

- 肩线是否平
- 上身是否明显歪斜
- 身体是否前倾或后仰
- 头肩是否稳定
- 近景时头肩姿态是否端正
- 全景时整体站姿是否正

用户侧文案例子：

- `当前上身整体比较正，镜头感稳定`
- `你现在更像是在歪着讲，画面会显得不够稳`

当前数据来源：

- 前端 pose
- 后端 pose rule

后续增强来源：

- Omni

#### C. 肢体松弛度与紧绷感

指标：

- 整体是否过紧
- 肩颈是否明显耸起
- 上身是否显得拘束
- 动作是否僵硬
- 是否有“收着讲”的感觉
- 是自然打开、平和，还是明显往下塌

用户侧文案例子：

- `当前肢体偏收，整个人会显得有些紧`
- `你的上身是打开的，镜头里的存在感比较好`

当前数据来源：

- Omni 为主
- Pose 只提供辅助信号

原因：

这类判断是“姿态 + 气质 + 连续画面”综合语义，不适合纯规则硬算。

#### D. 手势自然度与连贯性

指标：

- 是否有手势
- 手势是否过少
- 手势是否过多
- 手势是否和内容同步
- 手势是否自然
- 手势是否连贯
- 手势是否像“甩手”或“乱晃”
- 重点句是否有支撑动作

用户侧文案例子：

- `重点句里手势支撑偏少，表达会显得略平`
- `手势参与感不错，但略有点多，可以收一点`

当前数据来源：

- 前端 pose
- 后端 pose rule
- Omni 做自然度解释

#### E. 走位与空间使用

只在 `full-body` 模式下开放。

指标：

- 是否来回晃动过多
- 是否一直钉在原地
- 走位是否自然
- 走位是否服务于表达
- 转身、停步、移动是否突兀
- 空间使用是否积极

用户侧文案例子：

- `你的走位略显随机，观众会感觉你在晃`
- `这一段走位比较自然，能带动节奏`

当前数据来源：

- Full-body pose
- Omni

#### F. 表情与眼神互动

这是本维度里最晚成熟的一组指标。

指标：

- 表情整体基调：严肃 / 紧张 / 平和 / 积极 / 下沉
- 面部是否紧绷
- 眼神是否一直游离
- 是否有与观众互动感
- 是否一直在看屏幕或看稿
- 是否长期低头
- 是否有明显紧张感

用户侧文案例子：

- `眼神互动偏少，容易像在自说自话`
- `表情整体比较平，重点句可以再抬一点精神状态`

当前数据来源：

- Omni 为主

后续增强来源：

- face-specific 模型
- gaze 估计

### 4.3 维度状态如何产出

推荐规则：

- 如果关键证据不足，这一维度先不下结论，界面显示 `分析中`
- 如果有明确硬问题，例如离镜、严重偏心、明显歪斜，界面显示 `优先调整`
- 如果整体稳定但没有明显亮点，界面显示 `基本稳定`
- 如果肢体自然、镜头感稳定、互动感明显，界面显示 `做得好`

## 五、维度二：语音语调 & 节奏

这个维度负责回答：

- 你说得清不清楚
- 听起来稳不稳
- 重点有没有出来
- 节奏是不是舒服

### 5.1 一级子类

建议拆成 7 个子类：

1. 吐字与清晰度
2. 音量与投射感
3. 语速
4. 断句与停顿
5. 重音与强调
6. 起伏与情绪状态
7. 卡壳与流畅度

### 5.2 细粒度指标清单

#### A. 吐字与清晰度

指标：

- 吐字是否清楚
- 音节是否糊
- 句尾是否吞音
- 词边界是否清晰
- 快速表达时是否仍清楚

用户侧文案例子：

- `这段吐字偏糊，尤其句尾有点吞`
- `你的字头字尾比较清楚，信息传达效率高`

当前数据来源：

- Omni 为主
- ASR 错词率可做辅助信号

#### B. 音量与投射感

指标：

- 声音是否洪亮
- 声音是否偏小
- 是否显得虚
- 是否有过度喊的感觉
- 是否有“打到观众”的投射感

用户侧文案例子：

- `声音偏收，像只在对屏幕讲话`
- `当前声音比较能打出去，现场感不错`

当前数据来源：

- 音频能量特征
- Omni

#### C. 语速

指标：

- 语速偏快 / 正常 / 偏慢
- 语速是否稳定
- 是否在紧张时突然变快
- 是否在重点句前后能放慢

用户侧文案例子：

- `你这一段明显快了，听众会来不及消化`
- `当前语速控制得比较舒服`

当前数据来源：

- ASR + 时间戳
- Omni

#### D. 断句与停顿

指标：

- 断句是否自然
- 停顿是否正确
- 是否该停不停
- 是否不该断却断开
- 是否有“满句平推”问题
- 重点前是否有准备停顿

用户侧文案例子：

- `重点句前没留停顿，信息显得有点挤`
- `你的断句比较自然，听起来轻松`

当前数据来源：

- ASR 时间轴
- Omni

#### E. 重音与强调

指标：

- 关键字是否有重音
- 核心观点是否有强调
- 强调是否过度
- 是否所有句子一个强度
- 重点词是否被埋掉

用户侧文案例子：

- `核心词没有打出来，这句会显得不够有力`
- `你这句的重音落得比较准`

当前数据来源：

- Omni 为主
- 音频 prosody 特征可做后续辅助

#### F. 起伏与情绪状态

指标：

- 语气是平的还是有起伏
- 当前情绪状态是紧张 / 平和 / 积极 / 往下掉
- 是否有感染力
- 是否越来越没劲
- 是否明显发紧

用户侧文案例子：

- `语气整体偏平，重点句的起伏不够`
- `这一段明显更有精神，感染力上来了`

当前数据来源：

- Omni 为主
- 音频情绪特征做辅助

#### G. 卡壳与流畅度

指标：

- 是否频繁卡壳
- 是否有自我修正过多
- 是否重复起句
- 是否出现明显“找词”
- 是否存在长时间空白后硬接

用户侧文案例子：

- `这一段卡顿比较多，说明你在边讲边找词`
- `当前表达整体流畅，没有明显卡壳`

当前数据来源：

- ASR partial/final 行为
- 口头禅 / 填充词统计
- Omni

### 5.3 维度状态如何产出

推荐规则：

- 如果关键证据不足，这一维度先不下结论，界面显示 `分析中`
- 如果出现明显紧张、吐字糊、语速失控、频繁卡壳，界面显示 `优先调整`
- 如果节奏基本正常但没有明显亮点，界面显示 `基本稳定`
- 如果清晰、稳、起伏恰当、重点明确，界面显示 `做得好`

## 六、维度三：内容 & 表达

这个维度不评“知识对错”，而评：

- 表达是否清楚
- 内容是否简洁
- 结构是否清晰
- 是否有口头禅和废话

### 6.1 一级子类

建议拆成 7 个子类：

1. 简洁度
2. 口头禅与冗余
3. 重复与绕圈
4. 逻辑结构
5. 观点清晰度
6. 例子与支撑
7. 收束与推进

### 6.2 细粒度指标清单

#### A. 简洁度

指标：

- 句子是否过长
- 是否一句里塞太多信息
- 是否能快速说到点上
- 是否存在大量可删词

用户侧文案例子：

- `这段内容有点绕，删掉一层解释会更利落`
- `这一段很直接，信息效率高`

当前数据来源：

- ASR transcript
- Omni

#### B. 口头禅与冗余

指标：

- `嗯 / 哦 / 然后 / 就是 / 那个 / 其实` 等口头禅频率
- 重复修饰词
- 无效起句
- 句中填充词

用户侧文案例子：

- `“然后”“就是”有点多，会削弱专业感`
- `这段口头禅控制得不错`

当前数据来源：

- ASR transcript
- 规则统计
- Omni 做解释

#### C. 重复与绕圈

指标：

- 同一意思是否反复说
- 是否在原地绕
- 是否过度自我修正
- 是否信息推进太慢

用户侧文案例子：

- `你在重复同一个点，但没有继续推进`
- `这一段没有明显重复，推进比较顺`

当前数据来源：

- ASR transcript
- Omni

#### D. 逻辑结构

指标：

- 开头是否交代目的
- 中间是否有清晰层次
- 观点之间是否有过渡
- 结尾是否有收束
- 是否像“想到哪说到哪”

用户侧文案例子：

- `这段像在堆信息，层次还不够清楚`
- `你的结构比较清楚，听众容易跟住`

当前数据来源：

- Omni 为主
- transcript 分段规则可辅助

#### E. 观点清晰度

指标：

- 你的观点是否明确
- 核心句是否说透
- 是否一直在铺垫但没落观点
- 结论是否鲜明

用户侧文案例子：

- `你讲了很多背景，但观点还没真正落下来`
- `核心立场比较清楚，听众能快速抓住重点`

当前数据来源：

- Omni 为主

#### F. 例子与支撑

指标：

- 是否只有结论，没有支撑
- 是否有例子
- 例子是否贴题
- 是否有证据感

用户侧文案例子：

- `观点已经有了，但支撑例子还不够`
- `你这段例子接得比较好，能把观点立住`

当前数据来源：

- Omni 为主

#### G. 收束与推进

指标：

- 一句话说完后是否自然进入下一句
- 过渡是否清楚
- 一段讲完后是否有收束
- 是否能把听众往下带

用户侧文案例子：

- `段落之间的过渡有点硬`
- `这段推进顺，听众容易一路跟下来`

当前数据来源：

- transcript
- Omni

### 6.3 维度状态如何产出

推荐规则：

- 如果关键证据不足，这一维度先不下结论，界面显示 `分析中`
- 如果口头禅、重复、逻辑混乱明显，界面显示 `优先调整`
- 如果表达基本清楚但还不够利落，界面显示 `基本稳定`
- 如果结构清楚、表达利落、观点明确，界面显示 `做得好`

## 七、三大维度的数据来源建议

| 维度 | 当前主来源 | 当前辅助来源 | 后续增强 |
| --- | --- | --- | --- |
| 肢体 & 表情 | Pose Rule | Omni | Face/Gaze 模型 |
| 语音语调 & 节奏 | Omni | ASR 时间轴、音频特征 | 更细粒度 prosody 分析 |
| 内容 & 表达 | ASR Transcript + Omni | 规则统计 | 报告级结构分析 |

### 当前结论

V1 不建议把所有判断都交给 Omni。

最合理分工是：

- 几何硬信号：规则优先
- 语义解释：Omni 优先
- 内容表达：Transcript + Omni

## 八、刷新节奏设计

### 肢体 & 表情

- 更新频率：`1-2s`
- 数据窗口：最近 `2-3s`
- 特点：连续平滑，避免抖动

### 语音语调 & 节奏

- 更新频率：按一句话或最近 `3-5s`
- 数据窗口：最近 1 句或最近几秒
- 特点：要跟着自然停顿更新，不要每 500ms 刷

### 内容 & 表达

- 更新频率：最近 `1-2` 句
- 数据窗口：最近一个短段落
- 特点：刷新最慢，但最需要稳定

### 顶部主建议

- 更新频率：只有当“最值得纠正的问题”变化时才更新
- 作用：只做当前优先级最高的一条建议

## 九、每张卡应该怎么展示

### 卡片统一结构

每张卡只显示：

- 维度标题
- 用户可见状态标签
- 当前判断
- 当前建议

例如：

```text
肢体 & 表情
优先调整
当前头肩姿态不够稳定，镜头交流感偏弱
先把头肩位置定住，再继续讲重点句
```

### 用户可见标签建议

我建议统一用中文标签，不用英文，不用抽象词：

- `做得好`
- `基本稳定`
- `优先调整`
- `分析中`

其中：

- `做得好` 适合绿色
- `基本稳定` 适合中性色
- `优先调整` 适合橙红色
- `分析中` 不做强提醒色，适合浅灰或浅蓝

### “分析中”卡片怎么显示

这类卡片不要做成错误态，也不要做成告警态，而是一个温和占位：

```text
肢体 & 表情
分析中
正在更新这一项反馈
保持当前节奏
```

### 不应该展示什么

不应该在用户主面板展示：

- `centerOffsetX = 0.182`
- `shoulderTiltDeg = 8.4`
- `responseCount = 2`
- provider 名称
- 内部诊断字段

这些都放在内部诊断视图里。

## 十、系统降级策略

这是这版产品必须补的一层。

### 典型降级场景

#### 只露头

不能判断：

- 肩线是否倾斜
- 手势是否自然
- 上身是否稳定

应该输出：

- 内部：`insufficient_signal`
- 用户侧：`分析中`
- 文案：只显示通用占位态

#### 手不在画面里

不能判断：

- 手势自然度
- 手势是否过多

只能判断：

- 手势信号不足

#### 全身不可见

不能判断：

- 走位
- 空间使用

### 产品规则

任何维度下，只要关键证据不足，就必须允许内部进入 `insufficient_signal`。

但用户侧永远不要直接看到这个词。用户只看到：

- `分析中`

不要为了保持“卡片总有结论”而强行给用户：

- `做得好`
- `优先调整`

---

## 十一、建议的内部数据结构

为了支撑固定维度卡，后端最终不该只输出单条 `LiveInsight`，而应该输出：

```json
{
  "summary": {
    "title": "先把头肩位置定住",
    "detail": "镜头中的上身姿态还不够稳，先把头肩摆正，再继续讲重点句。"
  },
  "dimensions": {
    "body_expression": {
      "engine_status": "warning",
      "display_status": "优先调整",
      "headline": "头肩姿态还不够稳",
      "detail": "当前近景里你的头肩有些歪，镜头交流感会变弱。",
      "confidence": "high"
    },
    "voice_pacing": {
      "engine_status": "neutral",
      "display_status": "基本稳定",
      "headline": "语速基本正常",
      "detail": "但重点句的起伏还不够，核心词没有完全打出来。",
      "confidence": "medium"
    },
    "content_expression": {
      "engine_status": "positive",
      "display_status": "做得好",
      "headline": "内容主线比较清楚",
      "detail": "这一段表达没有明显绕圈，听众能跟住你的主线。",
      "confidence": "medium"
    }
  }
}
```

如果是待判断场景，应该长这样：

```json
{
  "body_expression": {
    "engine_status": "insufficient_signal",
    "display_status": "分析中",
    "headline": "继续观察中",
    "detail": "稍后更新这一项反馈。",
    "confidence": "low"
  }
}
```

这比当前单条事件流更适合产品 UI。

## 十二、推荐的开发顺序

### Phase A：先重构产品形态

先做：

- 右侧固定 3 维卡片
- 顶部 1 条主建议
- 去掉滚动 insight feed

不要先做：

- 新增更多模型
- 新增更多内部诊断字段

### Phase B：先把状态门补上

重点修：

- 只露头时不要判断肩线
- 手不入镜时不要判断手势
- 非全景时不要判断走位

### Phase C：再接维度化输出

把当前：

- `pose-rule`
- `omni-coach`
- `transcript`

统一汇总成三大维度状态，而不是直接把 provider 输出扔给 UI。

### Phase D：最后才补报告

报告页应复用这三大维度，而不是再发明一套新的评价体系。

---

## 十三、最终结论

`AI Live Coach` 的正确产品形态应该是：

- 顶部一条当前重点建议
- 中部三张固定维度状态卡
- 底部诊断区仅在内部联调时可见

固定三大维度：

1. `肢体 & 表情`
2. `语音语调 & 节奏`
3. `内容 & 表达`

而不是继续使用：

- 实时滚动提示流

这份设计的核心目标不是“让模型多说一点”，而是：

- 让用户在讲的时候，随时知道 **哪一类问题最值得先改**
- 并且看到一个**稳定、可持续跟踪**的训练状态面板
