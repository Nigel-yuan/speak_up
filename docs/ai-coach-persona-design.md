# AI 教练人格化设计

这份文档描述当前已经落地的“AI 教练人格化”方案，目标是把抽象的语音问答和同步分析，变成用户能明确感知的“正在陪我练习的教练”。

## 1. 产品目标

当前产品已经有四条核心能力：

- AI Live Coach
- AI 问答
- AI 训练报告
- 回放复盘

用户过去感知到的更像“系统能力”，而不是“具体的人在陪练”。当前已经解决的是：

1. 让用户在进入训练前明确选择自己的 AI 教练
2. 让训练中的右侧 coach 区真正显示“是谁在看我、是谁在点评我”
3. 让问答、报告、回放三个阶段都保持同一位教练的人设连续性

一句话目标：

**把 Speak Up 从“一个会分析的系统”，升级成“一个有形象、有口吻、有风格的 AI 教练”。**

## 2. 教练资料源

统一把教练资料收敛到 `ai_coach/`：

- `ai_coach/profile.md`
- `ai_coach/profiles.json`
- `ai_coach/avatar/*.jpg`

其中：

- `profile.md` 给人看，方便继续维护文案
- `profiles.json` 给前后端代码读，避免同一份人格配置重复写两套

## 3. 统一教练模型

每位教练至少有下面几类信息：

- 基础身份：`id / name / persona_type / avatar_file`
- 前端展示：`slogan / bio / live_status`
- 问答语音：`provider_voice_id / omni_voice_id / instructions`
- 报告风格：`report_style.instruction_zh`

所以“教练”不只是头像，而是一份贯穿全链路的统一 profile。

## 4. 交互改动

## 4.1 入口选择

训练入口改成先弹出 AI 教练选择层：

- 用户先选教练，再进入主训练页
- 弹层里展示头像、人格类型、slogan、简介
- 选中的 `coachProfileId` 会进入 session setup

入口不再暴露语言、场景和 voice 选项。场景默认走 `general` 通用表达训练，语言默认中文；训练模式在主页面顶部工具条里切换。

## 4.2 训练主页面

训练页仍然保持：

- 左侧：用户表达区域
- 右侧：coach 区域

右侧 coach 区域分成两块：

1. 实时文字稿
2. AI Live Coach

右侧 coach 区整体作为一个独立区域。

教练状态栏固定放在右侧顶部，也就是**实时文字稿上方**，主页面其余位置不再重复展示教练身份。

状态栏内容包括：

- 教练头像
- 教练名字
- 人格标签
- `实时状态表达`

这里不再展示 slogan，实时状态文案直接使用 profile 中的 `live_status`。只有训练开始后才对状态文案加扫光高亮，未开始时保持静态展示。

状态栏支持下拉切换教练，但只在 QA 未开启时可用。

## 4.3 问答模式

进入 QA 后：

- 右侧 Standby 头像切换成当前教练头像
- 问答语音使用当前教练 profile 对应的音色和口吻
- UI 上不再显示 voice profile 切换器
- QA 进行中不允许改教练

也就是说：

- 训练前可换教练
- QA 开始后锁定本轮教练

## 4.4 报告页

报告页补一块教练身份展示：

- 头像
- 名字
- 人设标签

报告文案生成时，还要把当前教练的人设限制传给后端，保证不同教练生成出来的语气有差异，但都必须保持“面向用户、可理解、不过度角色扮演”。

## 4.5 回放复盘页

回放页同样展示当前教练头像和名字，但不允许切换。

这样回放时，用户会继续感知到：

- 左边是自己当时的表现
- 右边是同一位教练当时给出的建议

## 5. 状态与锁定规则

教练切换规则统一如下：

- 训练未开始：可切换
- 训练进行中但未开启 QA：可切换
- QA 已开启：锁定，不可切换
- 报告页：只显示，不可切换
- 回放页：只显示，不可切换

## 6. 后端改动

后端要新增一个一等字段：`coachProfileId`。

它至少要进这几条链路：

- `POST /api/session/start`
- `SessionRecord`
- 报告上下文 `ReportJobContext`
- replay 返回结构
- report 返回结构

QA 问答链路里，`coachProfileId` 最终会映射到：

- provider voice
- omni voice
- interviewer instructions

报告链路里，`coachProfileId` 会映射到：

- 报告 system prompt / user prompt 里的风格约束

## 7. 技术实现原则

1. 不直接从 `profile.md` 解析业务逻辑
   - `profile.md` 是人读文档
   - `profiles.json` 才是程序配置源

2. 前后端共用同一份教练配置
   - 避免前端一套名字、后端一套名字、报告再一套口吻

3. 教练形象与语音保持同一个 id
   - 不再让用户感知“头像是一个人、声音又是另一套 profile”

4. 报告风格只能影响措辞，不影响评分标准
   - 分数仍然只由表现证据决定
   - 人设只影响表达方式

## 8. 落地顺序

当前已经按下面顺序落地：

1. 统一教练配置源：`profile.md + profiles.json`
2. 补前端教练选择与展示
3. 把 QA 的 voice profile 改成 coach profile 驱动
4. 报告页、回放页补充教练展示
5. 把报告生成 prompt 接入教练风格约束

## 9. 完成标准

当前用户应能明确感知到：

- 进入训练前，我选的是哪位 AI 教练
- 训练时，右侧是谁在陪我
- 问答时，说话的是同一位教练
- 报告和回放里，仍然是同一位教练

如果这四个感知能闭环，说明“AI 教练人格化”已经真正落地，而不是只换了一张头像。
